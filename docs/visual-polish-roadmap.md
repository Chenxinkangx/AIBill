# AIBill 后续视觉高级感优化计划

## 1. 背景

AIBill 已经完成了第一轮 shadcn/ui 方向的重构：项目中已经有基础 UI 组件、Tailwind token、Lucide 图标和更统一的弹窗/卡片/按钮基础。

下一阶段的重点不应该只是继续“替换组件”，而是把 AIBill 从一个可用的记账工具，推进到一个更有产品质感的移动端预算应用。

这份文档用于说明后续优化方向、执行顺序和验收标准。

## 2. 总体目标

后续视觉优化目标可以概括为：

> 建立 AIBill 自己的设计语言，让页面看起来更清爽、更有层次、更像一个成熟的移动端财务产品。

具体目标：

- 提升首页第一印象。
- 统一金额、预算、状态、卡片、表单的视觉表达。
- 减少页面里的零散 className。
- 让 shadcn/ui 成为底层零件，而不是最终视觉风格。
- 强化移动端 App 感。
- 保持轻量，不引入过重的 UI 库。

## 3. 视觉方向

建议 AIBill 采用：

> 轻量金融感 + 柔和数据卡片 + 克制紫色品牌感

关键词：

- 冷白背景
- 柔和紫色主色
- 数据卡片层次清晰
- 金额层级明确
- 成功、警告、危险状态统一
- 留白更大
- 阴影更克制
- 动效轻，不花哨

不建议走的方向：

- 大面积高饱和渐变
- 到处使用强阴影
- 每个页面各自一套颜色
- 复杂玻璃拟态
- 过度装饰图标

## 4. 推荐优化路线

建议分 6 个阶段推进。

```txt
阶段 1：升级全局视觉 token
  ↓
阶段 2：封装 AIBill 业务 UI 组件
  ↓
阶段 3：重做 Dashboard 首页
  ↓
阶段 4：优化底部导航
  ↓
阶段 5：统一表单体验
  ↓
阶段 6：优化设置页和记录页细节
```

## 5. 阶段 1：升级全局视觉 token

### 5.1 目标

先统一全局颜色、圆角、背景、状态色，让项目后续优化有一个稳定基础。

### 5.2 建议调整内容

修改 `src/index.css` 中的 token。

建议补充或强化：

- `background`
- `card`
- `primary`
- `muted`
- `success`
- `warning`
- `destructive`
- `border`
- `ring`
- `radius`

建议方向：

```css
--background: 更柔和的冷白色;
--card: 接近纯白但略带层次;
--primary: 稳定一点的紫色;
--success: 预算安全 / 收入;
--warning: 接近预算上限;
--destructive: 超支 / 删除 / 清空;
--radius: 0.875rem 或 1rem;
```

### 5.3 验收标准

- 页面整体不再像默认灰白后台。
- 卡片和背景能自然区分。
- 预算状态色在不同页面表达一致。
- 不影响现有构建。

## 6. 阶段 2：封装 AIBill 业务 UI 组件

### 6.1 目标

不要让页面直接堆 shadcn/ui 基础组件和大量 className，而是沉淀 AIBill 自己的业务组件。

### 6.2 推荐新增组件

```txt
src/components/common/
  PageHeader.tsx
  SectionCard.tsx
  MetricCard.tsx
  MoneyText.tsx
  StatusBadge.tsx
  FormField.tsx

src/components/ui/
  native-select.tsx
```

### 6.3 组件说明

| 组件 | 用途 |
| --- | --- |
| `PageHeader` | 统一页面标题、说明、右侧操作 |
| `SectionCard` | 统一业务区域卡片 |
| `MetricCard` | 金额、预算、收入、剩余等指标卡 |
| `MoneyText` | 统一金额颜色、字号、正负号 |
| `StatusBadge` | 统一预算状态、标签状态、AI 识别状态 |
| `FormField` | 统一 label、错误、描述、控件间距 |
| `NativeSelect` | 保留移动端原生选择器，同时统一样式 |

### 6.4 验收标准

- 主要页面里的重复 className 明显减少。
- 金额显示不再各写各的。
- 表单 label、错误提示、说明文字统一。
- 后续改视觉时优先改这些业务组件，而不是到页面里找 className。

## 7. 阶段 3：重做 Dashboard 首页

### 7.1 为什么先做首页

首页是产品第一印象，也是预算应用最核心的页面。

如果只能先优化一个页面，优先选 Dashboard。

### 7.2 推荐结构

```txt
MonthPicker
  ↓
主预算仪表卡
  - 本月预算
  - 已花
  - 剩余
  - 今日建议可花
  - 进度状态
  ↓
预算提醒卡
  - 未分配预算
  - 超支提醒
  - 本月收入提示
  ↓
分类预算网格
  - 分类圆环
  - 剩余 / 超支金额
  - 点击查看明细
  ↓
快速记一笔入口
```

### 7.3 视觉建议

主预算卡可以比普通卡片更高级：

- 使用轻微渐变背景。
- 大金额居中或左上突出。
- 进度条更粗一点。
- 已花 / 剩余 / 收入用小指标卡表达。
- 超支状态使用清楚但不刺眼的红色。

### 7.4 验收标准

- 打开应用第一眼有明显质感提升。
- 用户能快速看懂本月预算状态。
- 分类预算网格不拥挤。
- 移动端和桌面端都不显得空。

## 8. 阶段 4：优化底部导航

### 8.1 当前问题

底部导航已经替换为 Lucide 图标，但还可以进一步提升 App 感。

### 8.2 建议优化

- 背景使用 `bg-card/90`。
- 增加 `backdrop-blur`。
- 顶部边框弱化。
- 当前项增加柔和背景或小圆点。
- 中间“记账”按钮保留突出样式，但阴影更克制。
- 图标和文字大小固定，避免切换时抖动。

### 8.3 验收标准

- 页面切换时导航不漂移。
- active 状态清楚。
- 中间按钮突出但不突兀。
- 在 iOS/Android PWA 里底部安全区表现正常。

## 9. 阶段 5：统一表单体验

### 9.1 目标

记账页、编辑记录、设置页、AI 识别结果里的表单控件要统一。

### 9.2 优先文件

```txt
src/components/record/ManualForm.tsx
src/components/records/RecordActions.tsx
src/components/record/AiParseResultList.tsx
src/components/settings/AiConfig.tsx
src/components/settings/CurrencySetting.tsx
```

### 9.3 建议优化

- 使用 `FormField` 统一 label 和错误提示。
- 使用 `Input` / `Textarea` / `NativeSelect` 统一控件。
- 金额输入更突出。
- 类型切换使用 segmented control。
- 保存按钮使用更稳定的大按钮样式。
- 编辑表单减少拥挤感。

### 9.4 验收标准

- 新增记录和编辑记录的表单风格一致。
- 所有输入框高度、圆角、focus 状态一致。
- 移动端点击区域足够大。
- 错误提示清楚但不刺眼。

## 10. 阶段 6：优化设置页和记录页细节

### 10.1 Settings 页面

设置页建议改成成熟 App 的设置列表。

推荐结构：

```txt
偏好设置
  - 默认预算
  - 货币单位

AI 设置
  - 模型
  - API Key

数据管理
  - 导出数据
  - 导入数据
  - 清空数据

标签管理
  - 标签列表
  - 新增标签
```

每个设置项建议包含：

- 左侧图标
- 标题
- 描述
- 当前值或右侧操作

### 10.2 Records 页面

记录页建议重点优化：

- 筛选区域视觉。
- 记录卡片金额层级。
- 收入和支出颜色。
- 编辑/删除操作区。
- 空状态。

### 10.3 验收标准

- 设置页不像后台表单，而像移动端 App 设置页。
- 记录页列表信息密度合理。
- 用户能快速区分收入、支出、标签、日期。

## 11. 不建议现在优先做的事

以下事项可以做，但不建议排在最前面：

### 11.1 大规模动画

原因：动画能加分，但基础视觉还没完全统一前，先加动画容易显得花。

### 11.2 深色模式

原因：深色模式需要完整 token 和页面适配。建议等浅色主题稳定后再做。

### 11.3 重做所有页面

原因：一次性全改风险高，而且不容易判断哪一步带来提升。

### 11.4 引入另一个 UI 库

原因：当前 shadcn/ui 已经足够。继续引入 HeroUI、Ant Design 等会让视觉体系混乱。

## 12. 技术债与工程优化

### 12.1 包体积

当前构建仍有主 chunk 超过 500kB 的提示。

后续建议做：

- 路由级懒加载。
- AI 相关模块按需加载。
- 图标按需 import。

优先级：中。

### 12.2 文档同步

当前 shadcn/ui 组件底层可能使用 Base UI 或 Radix UI，具体以生成代码为准。

建议把文档里的“Radix”描述改成：

> shadcn/ui 负责生成可修改的组件源码，底层交互能力可能来自 Radix UI、Base UI 等 headless primitives，具体以当前生成代码为准。

优先级：中。

### 12.3 中文编码

如果终端里看到中文乱码，不一定代表文件内容损坏，但后续编辑时应统一使用 UTF-8。

优先级：中。

## 13. 推荐第一轮落地任务

为了最快看到“高级感”提升，建议第一轮只做这几件事：

```txt
1. 调整全局视觉 token
2. 新增 SectionCard / MetricCard / MoneyText / StatusBadge
3. 重做 Dashboard 首页
4. 小幅优化 BottomNav
5. 跑 npm run build 验证
```

不建议第一轮同时改 Records、Settings、AI 表单。先把首页做漂亮，确认视觉方向后再扩展到其他页面。

## 14. 验收清单

每一轮优化完成后，至少检查：

- `npm run build` 通过。
- 首页移动端显示正常。
- 底部导航不漂移。
- 弹窗圆角正常。
- 新增记账流程正常。
- 删除记录确认弹窗正常。
- 预算设置流程正常。
- 页面在桌面宽度下不显得过窄或过空。

## 15. 最终方向

AIBill 后续优化的关键不是“继续套 shadcn/ui 默认样式”，而是：

> 以 shadcn/ui 为底层零件，沉淀 AIBill 自己的预算、金额、状态和移动端交互组件。

当这些业务组件稳定后，项目的 UI 会越来越统一，也会越来越像一个真正成熟的产品。
